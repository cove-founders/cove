# Cove 产品准则：Input-First

本文档定义 cove 的核心产品准则，供团队讨论和对齐。所有功能设计决策应以此为约束。

---

## 1. 核心准则

**Input + One Click to View**

所有 cove 功能通过对话输入完成。用户说出意图，AI 执行，用户一键查看结果。

三条规则：

1. **输入即操作** — 用户通过自然语言描述意图，而非在 UI 中寻找按钮和菜单。
2. **一键看结果** — AI 产出的结果以符合人类视觉感知的方式呈现。文档以排版后的形态展示，图表可交互，数据以可视化方式呈现。用户点击即进入完整的视觉体验，无需多步操作。
3. **拒绝复杂配置** — 如果一个功能需要用户在多个面板之间切换、填写表单、或理解选项含义，它违反了 cove 的原则。

---

## 2. View 层

View 是 cove 的完整视觉体验层，不是技术意义上的 preview 组件。

### 定义

View 层的职责是让 AI 产出的结果以符合人类视觉感知习惯的方式呈现。用户看到的不是原始数据或源码，而是经过渲染、排版、可交互的最终形态。

- 文档以排版后的形态展示，而非 markdown 源码
- 图表可交互，而非静态图片
- 数据以可视化方式呈现，而非原始表格
- 幻灯片以演示形态展示，而非文本大纲

Preview（预览）只是 View 层的一种形态 — 在结果尚未最终确认前的快速查看。View 层的范围远大于 preview。

### 与传统软件的区别

传统软件的 view 是固定 UI 模板 — 开发者预先设计好页面结构，用户只能在预设的视图中查看数据。

Cove 的 View 层是 AI 驱动的动态视觉呈现。AI 根据产出内容的类型自动选择最合适的呈现方式，用户无需在「源码视图」和「渲染视图」之间手动切换。

---

## 3. 为什么

### 传统软件的问题

传统软件把内部结构暴露给用户。菜单层级反映的是代码模块划分，配置面板反映的是系统参数，多步操作流反映的是后端处理流程。用户被迫学习软件的组织方式才能使用它。

UI 是一种需要学习的抽象。用户并非天生就会使用下拉菜单、多级设置页、或 drag-and-drop。这些交互范式是被教育出来的 — 用户适应软件，而非软件适应用户。

### AI-native 的机会

自然语言是人类的原生接口。AI 的能力使得用户不再需要翻译自己的意图为软件能理解的操作序列 — 直接说出来就够了。

这带来三个结构性优势：

| 维度 | 传统 UI | Input-First |
|------|---------|-------------|
| 认知成本 | 用户需要知道「功能在哪里」和「怎么操作」 | 用户只需要知道「我想要什么」 |
| 功能扩展 | 每增加一个功能，UI 表面积增长（新菜单、新页面、新按钮） | 新增能力只需新的 skill/tool，产品表面积恒定 |
| 视觉体验 | UI 模板固定，view 由开发者预设 | View 层是产品体验的核心，AI 根据内容动态选择最佳呈现方式 |

---

## 4. 可发现性

Input-First 的最大挑战：用户不知道能问什么。

传统 UI 的一个优势是可见性 — 菜单和按钮告诉用户「这些是你能做的事」。纯对话界面中，这种可见性消失了。

以下四种机制按侵入性从低到高排列，共同解决可发现性问题：

### 4.1 AI 主动提示（最自然）

对话本身就是发现机制。AI 在完成任务后，根据上下文提及相关能力：

> 「文档已生成。如果需要，我也可以直接转成 PDF 或生成配套的演示幻灯片。」

不是每次都提示。系统判断用户尚未使用过某个相关能力时才触发。这比任何静态菜单都精准 — AI 知道用户的上下文、产品的全部能力、以及用户还没用过什么。

传统软件靠 UI 做发现。AI-native 产品靠 AI 自身做发现。

### 4.2 空状态引导

新对话的空白区域放置可点击的 prompt 示例。按场景分类，而非按功能分类：

- 「帮我写一份周报」而非「使用文档生成工具」
- 「分析这个 CSV 的趋势」而非「使用数据分析技能」

用户点击即开始对话。本质上仍然是 input — 只是降低了第一步的门槛。

### 4.3 半结构化入口（`/` 和 `@`）

用户输入 `@` 时展示可引用的工具、技能、文件列表。输入 `/` 时展示意图分类。

这是 Raycast 模式：搜索即发现，输入框本身就是探索工具。用户不需要离开输入框就能浏览全部能力。

### 4.4 结果卡片上的上下文动作

AI 返回结果时，结果卡片底部附带 1-2 个上下文相关的下一步按钮。比如生成了表格后，出现「导出 CSV」和「生成图表」。

这些按钮不是固定 UI 元素，而是 AI 根据当前结果动态生成的建议。仍然符合「one click」原则。

## 5. 功能设计约束

新功能上线前，用以下 checklist 检查是否符合 Input-First 准则：

- [ ] 用户能否通过一句话描述意图来触发此功能？
- [ ] 结果是否可以一键查看，无需额外导航？
- [ ] 是否避免了需要用户理解的配置项？（如果必须有配置，能否通过对话完成？）
- [ ] 新增此功能后，产品 UI 表面积是否保持不变？（无新菜单、新页面、新设置面板）
- [ ] 可发现性：AI 是否知道何时建议用户使用此功能？

### 合规示例

| 场景 | 做法 | 判定 |
|------|------|------|
| 用户想改 AI 的语气 | 在对话中说「以后回复用更简洁的风格」 | 合规 |
| 用户想生成文档 | 在对话中说「帮我写一份 XX 方案」，点击查看 | 合规 |
| 用户想调整模型参数 | 打开设置面板 > 找到模型 > 滑动 temperature 滑块 | 不合规 |

### 灰色地带

View 层的视觉呈现（文档渲染、图表交互、数据可视化）不是灰色地带 — 它是产品体验的核心部分。

灰色地带存在于操作过程中的交互。某些任务天然需要过程中的视觉反馈（如空间布局调整、颜色选择）。处理原则：

- 如果视觉反馈属于结果的呈现（View 层），直接支持，无需讨论
- 如果视觉反馈是操作过程的一部分（拖拽排序、手动调色板），需要评估是否有对话替代方案

## 6. 开放讨论

以下问题尚无定论，需要团队共同决策：

### Q1: 设置页面的定位

当前 cove 有设置页面（供应商配置、Skill 管理等）。是否应该将这些也迁移到对话中完成？还是将设置页面视为「底层基础设施配置」，允许其存在但限制其增长？

### Q2: 可发现性的优先级

四种可发现性机制（AI 主动提示、空状态引导、`@`/`/` 入口、结果卡片动作）中，应该优先投入哪些？是否有我们忽略的方式？

### Q3: 复杂任务的边界

当任务本身就很复杂时（例如用户需要逐步调整一个多页文档的格式），对话交互是否真的比直接操作更高效？边界在哪里？

### Q4: 指标

如何量化衡量 Input-First 准则的执行效果？可能的指标：
- 完成任务所需的平均交互步数
- 用户首次使用新功能的时间（从不知道到成功使用）
- 设置页面的访问频率（应该趋近于零）

### Q5: 与 CLAUDE.md 设计系统的关系

当前 CLAUDE.md 中的「Quiet Elegance」设计系统主要描述视觉层面。Input-First 是更高层的产品准则。二者的关系是什么？是否需要在 CLAUDE.md 中增加产品准则章节？
